Why the AI layer is empty

Root cause, traced through the code

collect_evidence() returns {"inventory": [...], "import_issues": [...]} and the whole dict is sent to the model. import_issues is the rule engine's output — the answer key. The model reads the nine issues, restates them, and merge_findings() then drops every one of them for colliding with a rule on (hardware_id, code).

evidence  = rules found 9 things  ──┐
                                    ├──> model repeats the same 9
prompt    = "here are those 9"    ──┘
                                          │
merge_findings() dedupes on (id, code) ───┘──> 9 dropped, 0 kept

You showed the model the answer key, then penalised it for copying. The instruction "look for concerns not already obvious" is not strong enough to overcome the fact that the obvious answers are sitting in its context window.

The fix, and why it is better than the original design

Stop sending import_issues to the model. Let it audit the raw inventory independently, then compare the two result sets:

  • Both found it → corroborated. Two independent methods agree. Highest confidence.
  • Rules only → rules_only. Provable, but the model missed it — worth knowing.
  • Model only → model_only. Genuine new signal a schema check cannot express. Needs human review.

The nine "drops" become nine corroborations — a real quality signal instead of wasted tokens — and anything the model finds beyond them is the actual value you are paying for. This is also the honest answer to "what is the AI actually for?"

What the backend gets genuinely right

Rent and return are correct under concurrency. BEGIN IMMEDIATE, then a conditional UPDATE … WHERE id = ? AND status = 'Available' AND is_damaged = 0, then a rowcount check. That is a compare-and-swap. Backed by a partial unique index on active rentals as a second line of defence at the database level. Two users clicking Rent simultaneously cannot both win.

The seed loader quarantines instead of overwriting. Duplicate id 4 is rejected and preserved as evidence, not silently reallocated. Original status is kept while is_damaged blocks the rental separately — source evidence intact, unsafe action blocked.

The auditor's trust boundary is right (once the two P0s land): rules are the floor, the model is additive, the wire schema is separate from the validation schema, unknown IDs are dropped and counted.

Grill

Answer out loud before opening each one. If you cannot, that part of the walkthrough will hurt.

Two people click Rent on the same laptop at the same millisecond. Walk me through what stops them both getting it.certain

Three layers. BEGIN IMMEDIATE takes SQLite's write lock, so the second request waits rather than reading stale state. Then the UPDATE is conditional — WHERE id = ? AND status = 'Available' AND is_damaged = 0 — so it only fires if the row is still available at write time. Then rowcount is checked: if it's 0, someone beat us, and we return 409 instead of pretending we succeeded. That's a compare-and-swap. Underneath, a partial unique index on rentals(hardware_id) WHERE returned_at IS NULL makes a second active rental physically impossible even if the application logic were wrong.

This is the race-condition question from the written assessment, answered in your own code.

Why check rowcount? You already checked the status two lines earlier.certain

Because the check and the write are two separate operations, and anything can happen between them. The read tells you what was true; the conditional update tells you what is true at the moment of writing. Checking status first only produces a nicer error message — it is the WHERE clause plus rowcount that actually makes it safe. Remove them and you have a time-of-check-to-time-of-use bug.

The seed has two records with id: 4. Why reject the second instead of giving it a new ID?likely

Because reallocating invents data. You don't know whether it's a genuinely different laptop that was mis-keyed, or a duplicate export of the same one. Silently assigning id 12 fabricates an asset that may not exist. Rejecting it and preserving the raw record in import_issues means a human can decide, and the evidence is still there when they do.

The general principle, and it's the one the whole task is testing: when data is ambiguous, quarantine and surface — never guess. Same reason 22-05-2023 is rejected rather than parsed: is that May 22nd, or a broken US date? Guessing is how you end up with a warranty claim on the wrong day.

Why is is_damaged a separate column instead of just setting status to Repair?likely

Because status is what the source file claimed, and is_damaged is what we concluded. The Dell says Available while its notes say "Battery swelling, do not issue." Overwriting the status to Repair would destroy the evidence of the contradiction — the auditor could never report it, because the conflict would be gone.

So: keep the claim, add the safety flag, block the rental on the flag, and let the auditor report the disagreement between the two. Preserve source, block unsafe action, surface the conflict.

Your AI auditor returned zero findings on the live run. Why?certain

Because I sent the model the rule engine's output as part of its input, then deduplicated its response against that same output. It re-derived the nine issues it had just been shown, and the dedupe dropped all nine. I gave it the answer key and then penalised it for copying.

The fix is to let the model audit the inventory independently, then treat agreement as corroboration rather than duplication — findings both methods catch are high-confidence, findings only the model catches are the actual new signal, and findings only the rules catch tell me where the model is weak.

Say this one out loud in the interview. Finding it required running the thing for real instead of trusting green tests.

Why do you validate the model's JSON with Pydantic when OpenRouter already enforced a schema?likely

Because the provider's guarantee is not my guarantee. Structured output means the shape is probably right; it says nothing about whether the content is true. The model can return a perfectly-shaped finding about hardware id 999, which does not exist. So there are two schemas doing two different jobs: a deliberately boring wire schema the provider can actually enforce, and a stricter Pydantic model that is my trust boundary. On top of that I check every hardware_id against the real source data and drop the ones that don't exist.

Model output is untrusted input. It gets the same treatment as a request body from a stranger.

Why is the audit a POST and not a GET?stretch

Because it isn't safe or idempotent in the HTTP sense. It costs money, hits a third-party provider, and returns something different each time. GET implies safe, cacheable, and repeatable — a proxy or a browser is entitled to prefetch it. You do not want your inventory audit silently re-billed because a crawler followed a link.

You validate password length in Pydantic on the server. The browser already enforces minlength="8". Why both?certain

Because the frontend runs on the attacker's machine and they control it completely. They can edit the JS, disable it, or skip the browser and hit the API with curl. Client-side validation is user experience; server-side validation is security. The server is the only thing I control, so it is the only place a rule is actually enforced.

Why PBKDF2 rather than SHA-256 for passwords?likely

Because SHA-256 is fast, and fast is exactly wrong here. An attacker with a leaked hash table wants to try billions of guesses per second, and SHA-256 lets them. PBKDF2, bcrypt, scrypt, and Argon2 are deliberately slow and tunable, which makes brute force expensive. Each password also gets a unique salt, so identical passwords produce different hashes and a precomputed rainbow table is useless.

If pushed: Argon2 is the modern default; PBKDF2 was chosen here because it's in Passlib with no native build step, which mattered on a five-hour clock. That's a trade-off I can name, not one I overlooked.

Why SQLite? When does it break?likely

Chosen because the whole app is one file, zero setup, and it deploys as a single service — correct for a five-hour build with an eleven-row seed. It breaks at concurrent writes: BEGIN IMMEDIATE takes a global write lock, so writers queue behind each other. Reads are fine; writes serialise.

The migration is Postgres with row-level locking, and the important part is that the rent logic doesn't change — a conditional update with a rowcount check is the same pattern there. Choosing compare-and-swap rather than an in-Python lock is what makes it portable.

Why not deploy on Vercel?stretch

Vercel is serverless: the filesystem is ephemeral and each invocation may get a fresh container. A SQLite file written during one request is simply not there for the next. Netlify Functions have the same property. Railway gives a persistent volume, which is what a file-backed database needs. It's not that Vercel is worse — it's that it's built for a stateless model my storage choice isn't compatible with.

Your delete endpoint returns 204. Why can't it return a body?stretch

204 means "No Content" — the spec forbids a body, and FastAPI enforces it. This actually broke during the build: the route was declared 204 while still returning JSON, and it failed immediately. It's in the commit history rather than rewritten away.

You used AI to write most of this. What did it get wrong, and how did you catch it?certain

Two answers, and the second one is the real one.

The small one: it declared the delete route as 204 while still returning a body. FastAPI rejected it immediately — caught by running the thing.

The one that matters: the auditor originally had LLM findings replace the deterministic ones, so switching the AI on silently discarded every finding the system could actually prove. Then, once that was fixed, the live run revealed the model was contributing zero findings because I'd fed it the rule engine's own output and then deduplicated against it. Both bugs passed every test — the tests were mocked, so they were testing my assumptions, not reality.

The lesson I'd give: green tests around an LLM call usually mean you tested the mock. The failure only appeared when I ran it against the real provider and looked at the numbers instead of the pass/fail.
