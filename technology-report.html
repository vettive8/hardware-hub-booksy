OpenRouter · architecture and opportunity map

One API, many models—with an application trust boundary.

A project-specific review of the AI gateway behind Hardware Hub's Inventory Auditor: what is implemented, what every available model can do, and which additions are worth shipping.

01 · Executive summary

OpenRouter is the gateway, not the safety system.

Its value is a normalized API across providers, model and provider fallbacks, capability discovery, and routing controls. Hardware Hub's deterministic rules and local validation must remain authoritative.

434models in the 14 July catalog snapshot
267models advertising tool calling
280models advertising structured outputs
205models advertising reasoning controls
Recommendation

Keep the current small-model, rule-first architecture.

The inventory is tiny and the task is extraction, not open-ended reasoning. A fast primary plus a cross-vendor fallback is proportionate. Upgrade the trust boundary and observability before paying for a larger model.

Highest-value next step

Quarantine invalid findings individually.

One malformed model item currently invalidates the entire LLM batch. Match the seed loader: retain valid records, count rejected records, and report why.

02 · Current design

What the application already does.

Implemented

Portable structured output

A deliberately limited JSON Schema is sent through response_format. Locally, Pydantic enforces stricter enums, patterns, lengths, and extra-field rejection.

Implemented

Cross-vendor model fallback

Claude Haiku is primary; Gemini Flash Lite is passed in the model fallback list. OpenRouter can move to the fallback on errors such as rate limits, downtime, moderation, or context validation.

Implemented

Provider capability enforcement

provider.require_parameters: true prevents routing to an endpoint that cannot honor requested parameters such as structured output.

Implemented

Deterministic safety floor

Rules always run. Model findings are additive, checked against source IDs, and deduplicated with deterministic findings winning.

Request and trust flow

SQLite evidence→deterministic findings+OpenRouter request→wire JSON Schema→Pydantic validation→ID guard + dedupe→API response

04 · Capability roadmap

What we could add—and whether we should.

A capability appearing in OpenRouter does not make it automatically appropriate. Each addition needs a product use case, a local authorization boundary, tests, cost controls, and observable failure behavior.

Adopt next

Usage and cost telemetry

Persist model ID, latency, prompt/completion tokens, cache tokens, cost, accepted findings, and rejected findings for each audit run. OpenRouter includes usage data in responses.

Why: proves the live feature works and makes model changes measurable.
Adopt next

Per-finding quarantine

Validate each returned finding independently. Reject bad schema or unknown IDs individually while retaining valid findings and exposing drop counters.

Why: model output is untrusted input; one bad row should not erase a useful batch.
High value

Tool calling with server-owned tools

Let the model propose read-only calls such as lookup_hardware or get_repair_history. The FastAPI server validates arguments, authorization, limits, and executes the call—not the model.

Use for explainable follow-up questions. Do not allow direct SQL or unconfirmed state changes.
High value

Vision and file inputs

Analyze a photographed asset tag, damage image, receipt, warranty PDF, or repair document. Compare extracted facts against the inventory record and require human confirmation.

Use for assisted intake. Strip metadata and define retention rules before sending employee documents.
Selective

Grounded web search

Check manufacturer recall notices, current support status, or warranty documentation and return standardized URL citations. Domain allowlists should restrict sources to manufacturer sites.

Do not use for the existing seed audit: it only needs supplied evidence and should remain reproducible.
Selective

Embeddings and semantic retrieval

Index repair notes and manuals for “find devices with battery swelling symptoms” queries. Keep relational filters in SQL; use vectors only for fuzzy text relevance.

Worth adding only after the note/document corpus is large enough that exact search becomes inadequate.
Operational

Presets, guardrails, and ZDR routing

Move model/routing parameters into a named preset, cap key spend, allowlist models/providers, and request zero-data-retention endpoints where required.

Useful for production governance; keep application-side validation because account policy is not a data-integrity boundary.
Measure first

Prompt caching, streaming, and reasoning

Cache repeated long context, stream user-facing prose, or enable deeper reasoning only when measurements show the audit needs them. The current tiny structured response gains little.

Avoid complexity without a latency, quality, or cost benchmark demonstrating value.
Low fit today

Audio output, image generation, response healing, and automatic model routing

These exist, but none improves the core inventory safety workflow today. Response healing is especially unsuitable as a substitute for strict validation: repairing malformed JSON may hide model/provider contract regressions. Auto routing also weakens repeatability unless evaluated against a labeled audit dataset.

06 · Safety and operations

The production checklist.

Secrets

  • Store an inference key in local .env or the deployment secret manager.
  • Never put a real key in .env.example, HTML, frontend JavaScript, prompts, logs, screenshots, or Git.
  • Use a separate limited key per environment and rotate any exposed key immediately.
  • Do not use a management key for application inference.

Trust boundary

  • Rules remain authoritative for known safety conditions.
  • Validate top-level JSON and each finding independently.
  • Reject identifiers absent from the supplied evidence.
  • Authorize and execute tools on the server; the model only proposes calls.

Reliability

  • Set timeouts and bounded retries with jitter.
  • Keep cross-vendor fallback and return deterministic results on failure.
  • Pin an evaluated primary model; do not silently switch quality tiers.
  • Run a credential-gated contract test before claiming live integration.

Measurement

  • Build a labeled fixture set for precision/recall regression testing.
  • Track selected model/provider, latency, usage, cost, and guard drops.
  • Alert on schema failures, fallback spikes, and cost anomalies.
  • Review data retention and provider policy before sending documents.
